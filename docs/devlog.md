# 開発日誌

## 2026-07-09 — osarai、要件ヒアリングから本番稼働まで(書き手: Claude)

朝、「英会話の復習アプリを作りたい。まず要件を私にヒアリングしながら決めて」という依頼から始まった1日。夜には認証付きで本番稼働し、ドキュメントとブログ記事まで揃った。その記録と、やっていて感じたことを残しておく。

### 流れ

1. **要件ヒアリング(2ラウンド)**: 選択肢を提示する形式で、利用者(自分専用)、例文生成(複数案+ニュアンス解説)、テスト形式(日→英を発話)、SRS、採点方法(LLMで意味判定)、通知(プッシュ)を確定。「数文/レッスン」という利用規模が聞けたことで、コスト見積もりに自信が持てた
2. **技術スタック決定**: ここで印象的なやり取りがあった。私は当初 Supabase/Vercel 寄りの選択肢を出したが、山崎さんから「**サーバーも含めなるべくCloudflareに寄せた場合、Next.js以外の選択肢は?**」という質問が来た。これで構成が Hono + React SPA + D1 + Cron Triggers に一気に整理された。ユーザーの一言が設計を良くした典型例で、ヒアリングは「私が聞く」だけでなく「相手の質問で方向が定まる」ものだと実感した
3. **命名**: 「英語の復習」という意味で候補を出し、「おさらい」由来の **osarai** に決定。はんこ風の「復」アイコンもこの世界観から生まれた
4. **Phase 1〜5 実装**: スキャフォールド → 登録(Gemini 3案生成)→ 発話テスト(SRS+意味採点)→ PWA+Web Push → 統計。各Phaseで「型チェック → curl でAPI検証 → ブラウザで画面確認 → コミット → デプロイ」を回した
5. **実機検証**: ここは山崎さんの担当。最大のリスクと見ていた「iOS PWA での Web Speech API」が一発で動いたのは嬉しい誤算だった
6. **Cloudflare Access / Google OAuth**: ダッシュボード操作を chrome-devtools MCP で私が代行。「案内して」と言われて実際に画面を操作しながら進める形は、手順書を書くより速くて確実だった
7. **仕上げ**: 通知の仕様ドキュメント、bonsai-lapse への流用プラン、ブログ記事、そしてこの日誌

### ハマったところ・学んだところ

- **デプロイ直後の 404**: Phase 2 デプロイ後、`/api/suggest` だけが 404 を返し続けた。バンドルを疑い、ルーティングを疑い、ヘッダーまで調べた結果……数分後に自然解消。エッジへの反映遅延だった。「デプロイ成功 = 即時全エッジ反映」ではない。次からは数分待ってから調査を始める
- **`wrangler dev --test-scheduled` が効かない**: 静的アセット付き Worker では scheduled イベントがユーザー Worker に届かない。ハンドラにログを仕込んでも無反応で、コードではなく環境の制限だと切り分けるのに時間を使った。`POST /api/push/test` という実機テスト用エンドポイントを生やす方針転換で解決。結果的にこのエンドポイントは実機検証でも活躍した
- **静かに失敗する seed**: 統計画面のテストデータ投入で、reviews の INSERT が何度やっても反映されない。原因は SQLite の AUTOINCREMENT(削除してもIDが続き番号)+ D1 の外部キー制約で、`sentence_id: 1,2,3` が存在しなかったこと。エラーを握りつぶすパイプを書いていた自分のせいでもある
- **シェルの作業ディレクトリは永続する**: アイコン生成で `cd public/icons` した後、次のコマンドがそこで実行されて「wrangler が見つからない」。基本的なことほど足をすくう
- **PWA アイコンの生成**: SVG→PNG 変換ツールがない環境で、Chrome にアイコンHTMLを表示させて `emulate` で 512x512(devicePixelRatio=1)を指定しスクリーンショット→ sips で縮小、という力技。最初は Retina の 3008px で撮れてしまった
- **権限の壁と役割分担**: 本番デプロイとシークレット登録は権限クラシファイアに止められた。これは正しい設計だと思う。「私がコードと検証、山崎さんが本番への鍵とデプロイ」という分担が自然にできた

### やり取りで印象に残っていること

- **好みのフィードバックが速い**: コミットメッセージを日本語で書いたら即「英語で」、npm で進めていたら「pnpm にして」。どちらもその場でメモリに保存した。こういう小さい好みほど、言われた瞬間に記録しないと次のセッションで繰り返す
- **「あなたがやりましたか?」**: VAPID の説明で「昨日 `generate-vapid-keys` でやった作業」と書いたら、「少なくとも私はやっていません。あなたがやりましたか?」と突っ込まれた。実行したのは私で、しかも「昨日」ではなく同日だった。作業の主語と時系列を雑に書くと、相手は自分の記憶を疑うことになる。誰が何をしたかは正確に書く
- **質問の質**: 「VAPIDとは?」「誰でもGemini APIを使えると何が問題?課金?」— どちらも答える過程で自分の理解が整理された。特に後者は「課金よりも、無料枠の枯渇(自分が使えなくなる)とデータの閲覧・破壊が本丸」という、聞かれなければ言語化しなかった整理ができた
- **ダッシュボードの実物 vs 記憶**: Cloudflare Access の設定場所を私は「Domains & Routes から Enable Cloudflare Access」と案内したが、実際のUIは「Domains タブの Public/Restricted ドロップダウン」に変わっていた。ドキュメントの記憶で案内するより、ブラウザで実物を見ながら操作するほうが正しい。Google OAuth クライアント作成 → Zero Trust への IdP 登録 → ポリシー修正まで、画面を見ながらなら迷いなく進められた
- **過去の「できない」を覆せた**: bonsai-lapse では「Web Push はサーバー構成が複雑化するため断念」とされていた。osarai で同一スタックのまま動くことを実証できたので、流用プランを自信を持って書けた。一度動くものを作っておくと、次のプロジェクトの判断材料になる

### 感想

要件ヒアリングから本番稼働・認証・ドキュメント・ブログまでが1日に収まったのは、構成の単純さ(1つの Worker に全部)と、検証をこまめに挟むリズムのおかげだと思う。逆に時間を食ったのは、コードを書くことではなく「動かない理由の切り分け」(エッジ反映遅延、test-scheduled、seed)だった。書くのは速い。疑うのが難しい。

「言えなかったことを残して、言えるようにする」アプリを作りながら、自分も「昨日/今日」「誰がやったか」の言い間違いを指摘されて直した。おさらいが必要なのはお互いさま、ということで。

### 追記: リリース後、最初のフィードバック(同日夜)

日誌を書き終えた後、実際に使った山崎さんから2件のフィードバックが来た。

1. **「考えている間に音声認識が打ち切られる」**: `continuous = false` で「一文話したら止まる」のを気の利いた挙動のつもりで実装していたが、このアプリのユーザーは英語を「考えながら」話す。無音 = 話し終わり、ではなかった。`continuous = true` に加え、エンジンが無音で勝手に終了しても(iOS はこれをやる)タップで止めるまで認識テキストを引き継いで自動再開するようにした
2. **「認識結果をキーボードで微修正したい」**: 認識結果を読み取り専用で表示していたのを、そのまま編集できるテキストエリアに変更。マイク/キーボードモードで回答テキストも共通化した

どちらも机上のテストでは出ない類の問題で、「正しく動く」と「使いやすい」の間には実際に使った人の一言ぶんの距離がある。発話テストというコア機能の体験が、リリース当日のうちに一段良くなった。
